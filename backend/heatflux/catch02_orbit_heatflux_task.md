# CATCH02 轨道外热流复现任务

## 1. 任务目标

基于《CATCH02 卫星整星正样设计报告》5.8 节，分别使用 **STK、GMAT、Orekit** 对 CATCH02 的轨道光照与外热流环节做一次复现分析，产出与报告中 **表 5-6、表 5-7、图 5-5 至图 5-8** 对应的信息。

本任务的目标不是做完整热设计，而是复现报告中的轨道外热流输入：用边长 1 m 正方体代表卫星，计算不同轨道光照工况下六个面的平均吸收热流密度和晨昏轨道四季瞬态外热流曲线。

## 2. 报告输入与待复现对象

### 2.1 基础输入

- 卫星：CATCH02
- 简化热模型：边长 `1 m` 的正方体
- 表面太阳吸收率：`alpha = 0.2`
- 表面红外发射率：`epsilon = 0.8`
- 分析日期：
  - 春分：3 月 21 日
  - 夏至：6 月 21 日
  - 秋分：9 月 21 日
  - 冬至：12 月 21 日
- 轨道光照类型：
  - 晨昏太阳同步轨道
  - 正午太阳同步轨道

### 2.2 轨道口径注意事项

报告前后存在轨道描述不一致：

- 第 3 章约束描述为：`600 km 太阳同步轨道`
- 第 5.2 节基准轨道描述为：`600 km、29°倾角、偏心率 < 0.003`

执行时请明确记录采用的轨道口径。建议：

1. 主复现按报告 5.8 节文字中的“晨昏/正午太阳同步轨道”建立工况。
2. 同时在说明中标注报告 5.2 节 `29°` 倾角与太阳同步口径不一致。
3. 如时间允许，增加 `600 km、29°、e=0 或 0.001` 的敏感性对比；若不做，也必须说明未做原因。

## 3. 三套工具执行要求

### 3.1 STK

目标：建立 CATCH02 轨道光照场景，导出或计算六个面的外热流输入。

要求：

- 建立 `600 km` 晨昏太阳同步轨道与正午太阳同步轨道工况。
- 覆盖春分、夏至、秋分、冬至四个日期。
- 导出太阳矢量、地影状态、地球可见性、卫星姿态/面法向相关数据，或直接导出可用于热流统计的数据。
- 生成表 5-6、表 5-7 对应的平均热流结果。
- 生成图 5-5 至图 5-8 对应的晨昏轨道瞬态曲线数据。

风险提示：

- Linux 服务器上不一定具备 STK 桌面版、STK Engine 或有效许可证。
- 如无法运行 STK，应在结果中明确说明环境限制，并保留 STK 任务脚本、接口调用尝试记录或 Windows/STK 替代数据来源。

### 3.2 GMAT

目标：用 GMAT 复现轨道传播、太阳方向、地影进出和轨道光照几何，为热流计算提供输入。

要求：

- 建立 `600 km` 近圆轨道。
- 分别配置晨昏太阳同步与正午太阳同步工况。
- 每个日期至少传播一个完整轨道周期，建议覆盖 24 小时。
- 输出卫星位置速度、太阳方向、地影状态、轨道时刻。
- 若 GMAT 不直接输出完整外热流，使用 Python 后处理计算六个面的太阳直射、地球反照、地球红外与总吸收热流。

### 3.3 Orekit

目标：用 Orekit 建立可在 Linux 命令行重复运行的开源基准计算。

要求：

- 使用 Java 或 Python/Orekit 均可，优先保证可复现。
- 配置并记录 Orekit data 目录来源。
- 建立同 GMAT 一致的轨道与日期工况。
- 计算太阳方向、地影状态、轨道位置、各面入射角或面法向投影。
- 用同一套 Python/Java 后处理模型输出平均热流表和瞬态曲线。

## 4. 热流计算模型要求

至少包含以下三类外热流：

- 太阳直接辐射
- 地球反照
- 地球红外辐射

可采用工程简化模型，但必须在 `README.md` 中写明：

- 太阳常数取值
- 地球反照率取值
- 地球红外等效辐射取值
- 地影判据
- 六个面的姿态定义
- 正方体坐标系与报告 `+X/-X/+Y/-Y/+Z/-Z` 的对应关系
- 平均值统计时间范围

## 6. 验收标准

- GMAT 与 Orekit 部分能在 Linux 服务器上重新运行并生成 CSV 与 PNG。
- STK 部分如无法运行，必须有清楚的环境限制说明；如可运行，应输出同样的 CSV 或中间数据。
- 至少产出报告表 5-6、表 5-7 的复现表格。
- 至少产出报告图 5-5 至图 5-8 的晨昏轨道四季瞬态外热流曲线。
- 图表必须包含标题、坐标轴、单位、图例。
- 结果说明必须记录与报告值的偏差，并说明可能原因，如轨道口径、姿态假设、地影模型、反照模型、红外模型或热流简化差异。
